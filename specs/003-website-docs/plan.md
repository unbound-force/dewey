### Writing Style Reference

All content must follow the established website conventions:
- **Frontmatter**: title, description, lead, date, weight, toc: true
- **Tone**: Direct, declarative, no hedging, no filler, no emojis in prose
- **Person**: Second person ("you") for guides, third person for team/project pages
- **Structure**: H2 for major sections, H3 for subsections, no H1 in body
- **Code blocks**: Language-tagged (`bash`, `yaml`, `json`, `text`)
- **Tables**: For comparisons, command references, structured data
- **Paragraphs**: Short (2-4 sentences)
- **Cross-links**: Every page ends with "Next Steps" section (2-4 bulleted links)
- **No superlatives**: Never "best", "revolutionary", "powerful"

## Coverage Strategy

This is a documentation-only feature that modifies Markdown files in an external repository (`unbound-force/website`). The constitution's Testability principle (§IV) requires features to include tests, but this principle is designed for production code changes, not static content.

**Justification for manual verification**: Static Markdown files in a Hugo site cannot be unit-tested in the traditional sense. The verification strategy substitutes automated code tests with:

1. **markdownlint** — automated lint checks catch formatting and structural issues
2. **Hugo build** (`hugo --minify`) — automated build verification catches broken frontmatter, invalid templates, and missing referenced files
3. **Link validation** — automated check that all internal cross-links resolve
4. **Content accuracy review** — manual cross-reference of documented commands/flags against `dewey --help` output (Task T027)

This approach aligns with the constitution's intent (verifiable quality) while acknowledging the limitations of testing static content. The primary risk — content inaccuracy — is mitigated by Task T002 (capture Dewey v0.2.0 CLI help output as a reference artifact) and Task T027 (cross-reference all documented commands against that reference).
